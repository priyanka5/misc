Our company is TrackingSport (www.trackingsport.com) and what we do is to provide support for the timers in sporting events. We are not a timekeeping company, but we provide services to the timekeepers so that they can publish online to Internet the times of passage through the control points. I do not know if you know anything about timing, so I'll try to explain it in a few simple words, although you can look at the websites of companies that provide timing tools such as Chronotrack or Mylaps, which are sure to explain it better. I tell you, the athletes have placed a passive disposable chip stuck on the bib that identifies them. At several control points, antennas or mants are placed that activate the chip when it passes over it. The chip responds with a signal and the system records the moment in which that signal has been produced (in fact it is more complex, but in a simple way it is like that). We are connected in a network with the timing system that sends us the time of passage of the chip, we process that chip and that time to assign it to an athlete and we publish on our web page the passage of the athlete. We also publish it in our mobile app and make a publication on the athlete's wall on facebook and twitter (previously they must have registered, of course). This system is complemented by an enrollment platform and an exclusive application for managing the profile of the athlete where they can see the results in all events, they can upload the documents needed to register for an event (medical certificate, parental permission, etc.). We use keycloak to access the registration portal and profile management. We have recently added another product to our system (www.pul-sos.com, although the web is not yet fully translated into English yet). It is a bracelet that athletes (or any other person, we have in fact provided coverage to an Alzheimer's disease association) for their safety. It is a silicone bracelet that has a unique QR code engraved. The athletes enter their profile and can incorporate the medical or emergency data they want. In case of accident, loss or any circumstance in which the person is not able to respond, the paramedics (or anyone who can help them) can read the QR and a page with all the data of the person is presented. Here they can indicate previous illnesses, allergies, contact persons, etc. The access to this portal for managing the profile and all the data that the user wants to appear when someone reads the QR is controlled by keycloak.

We are primarily using Keycloak to provide the Oauth2/OIDC authentication functionality for our microservice platform. We are not using it as the main repo of our user base, but only as a shadow of our user store in order authenticate users and provide bearer tokens, so we have implemented a few customizations in order to support our needs. Probably our most useful customization is that since our user interface is a React-based SPA, the requirement to do full-page redirects when authenticating a user is really antithetical to how an SPA needs to function. Not only does it require page changes, but is really limiting from a design and branding perspective since our customers will not have access to the keycloak server to customize themes. It also seems a little circa 2003. So I implemented a custom freemarker-based theme that returns json instead of the html. This allows our SPA to be able to better use and control that flow of data from Keycloak while still handling the redirects as needed and our customers will be able to implement their own UI designs without having to make changes to the Keycloak server. A suggested enhancement would be to have an option to return actual json rather than having to shoehorn it into a freemarker templates. Additionally, we support SMS based 2fa and because of the aforementioned use of an SPA, all of our multifactor support has to happen within our SPA, so we cannot use any of the Keycloak multifactor features since they are so ingrained with the redirect-based authentication flow. We are using many Keycloak realms to support our multi-tenant needs and Keycloak clients to represent our high level resources. I think the most challenging thing about Keycloak is that many of the features related to clients and authentication flows are not very intuitive and took a lot of work to understand just how they function and their limitations. We also tried to user the authorization features of Keycloak for resource protection, but we found them to be rather limiting and not at all friendly to use. So we only use Keycloak for user role assignments and then do the authorization of roles-based access externally, currently using Apiman with a custom plugin for supporting finer grained permissions. The authorization aspects of Keycloak while broad still feel very 1.0ish and not very useful. I do appreciate the availability of the javascript-based authentication module. Though I have built some custom SPIs, the availability of the script module has dramatically increased our efficiency and flexibility for being able to quickly make changes. Two suggestions for that script module would be to allow access to the HTTP response so that we can set our own application cookies or other custom headers and it would be great if there were built in objects to make restful calls from within the script that could be integrated with the Keycloak authentication functionality for passing bearer tokens, etc. At the moment, that all has to be done using custom javascript which is tedious and prone to error, but it seems like such a waste to not be able to take advantage of all of the Resteasy functionality that already exists under the covers.
